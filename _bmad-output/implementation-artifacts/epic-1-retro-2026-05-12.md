## Lo Que Necesita Mejorar

### 1. Ausencia de diseño UX (hallazgo más crítico — Santiago)

El epic completo fue backend. En cuanto toquemos frontend (Story 2.2), no existe ningún documento de UX: ni wireframes, ni flows de usuario, ni sistema de componentes, ni estados de UI (loading/error/vacío). El prototipo React+Konva en `src/` existe pero no es diseño — no tiene guía de colores, tipografía ni comportamiento de interacción documentado.

**Impacto directo:** Story 2.2 (Exploración y Selección de Prenda) no puede implementarse correctamente sin saber cómo debe verse y comportarse el catálogo.

### 2. Seguridad no detectada en implementación inicial

Story 1.7 tuvo 3 hallazgos de seguridad en el code review que no fueron identificados durante la implementación:
- SVG XSS almacenado (vector de ataque real si se almacena en MinIO y se sirve públicamente)
- Extensión de archivo inyectable vía `SanitizeFileName`
- IP rate limiting que no funcionaba detrás de proxy

La implementación pasó todos los tests pero tenía vulnerabilidades. El code review fue la única red de seguridad.

### 3. Atomicidad Redis no considerada en diseño inicial

Dos operaciones Redis en Story 1.7 (CreateTokenAsync y ConsumeTokenAsync) se implementaron como secuencias GET+DEL+SET no atómicas — TOCTOU real bajo concurrencia. Solo detectado en code review. El patrón correcto (Lua script atómico) no es obvio y requiere conocimiento específico de Redis.

### 4. Gaps de production-readiness

- Bucket MinIO solo creado en `IsDevelopment()` — habría fallado en cualquier entorno no-dev.
- Configuración MinIO sin validación al startup — la app hubiera arrancado silenciosamente sin MinIO y fallado en runtime.

Ninguno de estos fue detectado en la implementación ni en los tests de integración (que corren contra un entorno de test, no de producción).

### 5. Gap de tracking de progreso (Story 1.5b)

La implementación de Story 1.5b existía completa en el codebase pero la story nunca fue marcada como completada. El estado del epic era opaco: "aparentemente en progreso" cuando en realidad estaba hecho. Cualquier herramienta de tracking hubiera reportado Epic 1 como incompleto.

## Lecciones Aprendidas

1. **El backend puede avanzar sin UX; el frontend no.** El momento de diseñar la UX es *antes* de la primera historia de frontend, no durante o después.

2. **El code review es la única red de seguridad para vulnerabilidades.** Sin una fase de revisión adversarial, los problemas de seguridad pasan como implementación correcta porque los tests funcionales no los detectan.

3. **Redis = concurrencia. Concurrencia = atomicidad.** Cualquier operación Redis que involucre más de una llamada (GET + SET, GET + DEL + SET) es un TOCTOU potencial. El patrón por defecto debe ser Lua script atómico.

4. **"Funciona en dev" ≠ "funciona en producción".** Los tests de integración validan lógica de negocio; la production-readiness requiere un checklist explícito.

5. **Si no está marcado, no está hecho.** El código no comprometido en el story file es código invisible para el proceso. La disciplina de tracking es tan importante como la disciplina de código.
